# scratch

## technical

활동의 지리 정보는 다음과 같은 이유로 매우 중요

기본 활동 문제점 

- 중복 지역 활동이 있으므로, 지리 데이터를 활용하여 중복을 미연에 방지하도록 해야하는 목표 있었음

- 이와 반대로 소외된 지역에 대한 시각적 정보를 제공해야 하는 과제가 있었음

- 또한 기존의 지역 정보 등록이 매우 불편하고 정확하지 않았음 (수기)

주요 목표

- 2초 이내에 지도 상의 조건을 만족하는 활동 지표가 조회되어야 함

- 중복을 피하고 소외 지역을 발굴해내기 위한 시각적 지표가 있어야 함

- 기존에 존재하는 수기 데이터와 통합이 가능한 데이터 구조 설계 요구됨



레퍼런스로 oceanconservancy 사이트 참고함

북미를 중심으로 전 세계의 해양 폐기물 수거 활동을 열람할 수 있었음

전 세계 지도 위에 활동 표시되는것을 확인할 수 있었고 클러스터링 지원

허나 지도를 줌인 줌 아웃 할때마다 조회가 다소 늦어지는 (렉걸림) 현상 있었음



보다 효율적인 지리 정보 저장 및 활용을 위해 PostgreSQL의 라이브러리인 postgis 활용



postgis에 대한 내용, spacial indexing 학습함



위경도 기반 데이터와, 주소 기반 데이터를 모두 수용할 수 있고 통합된 하나의 데이터로 관리해야하는 요구사항



kakaomap api의 geocoding 활용하여 양방향 translation 가능하도록 함



또한 시,도 및 구 (region_1depth_name, region_2depth_name) 및 날짜를 인덱싱하여 특정 지역 및 날짜 조회 성능 개선. 특히 성능 분석 상에서 long tail 현상의 개선을 확인 할 수 있었음.



가중치 기반 히트맵 데이터 생성: 구글 지도 내장 기능으로 특정 점의 좌표와 그것의 weight(가중치)를 데이터로 주면 그에 맞게 히트맵을 오버레이해주는 기능이 있음. 이것을 활용하여 활동의 지역 불균형을 해소하기 위한 시각적 자료를 제공하고자 하였음. 데이터를 가공하기 위해 postgis 내부 함수 (st_snaptogrid 등) 으로 약 10km x 10km 으로 나누어진 각 그리드 안의 활동 횟수 합을 구하고 (group by) 이를 1에서 10사이로 정규화한 데이터를 가공하였음. 



사용자가 줌인/줌 아웃 시에 화면에 노출되는 지역의 지리 데이터만을 조회하는 bounding box 기법을 활용하여 ux을 향상시켰음. 이것 또한 postgis 함수 st_makeenvelope를 활용하여 구현하였음



지리 데이터 validation으로 한반도를 벗어난 데이터(잘못된 값) 입력을 원천 차단. 



해결하지 못한 잠재적 문제점:

geocoding 과정이 외부 api를 이용하여 게시글 저장에 지연이 있었음 -> async로 이를 해결할 수 있었으나 데이터의 무결성 깨질 위험에 노출



또한 외부 api에 의존하므로 서비스 자체의 불안정성 증가



해결방법으로 우리나라 지역 분할의 polygon기반으로 직접 geocoding 시도하였으나, 이 즈음 프로젝트가 마무리되어 완성 단계에 이르지는 못하였음.

코딩 컨벤션을 어떻게 지켜나갈것인가?

팀 프로젝트에 있어서 지켜야 하는건 마음속으로는 알지만 사실 지키기 상당히 쉽지않은 부분인것 같다. 띄어쓰기를 더 한다던가 변수 이름을 조금 다른 방식으로 선언한다고 앱에서 오류가 나는게 아니므로. 

그럼에도 일관된 코딩 컨벤션은 필수적이라고 느낀 계기 -> 필연적으로 다른 사람이 작성한 코드를 내가 읽어야 할 일이 생기고, vice versa. 그러므로 상세한 설명이 달린 주석쓰기도 매우 중요하다

PA + PostGIS 통합
JPA 환경에서 PostGIS를 사용하기 위한 세 가지 문제를 해결했습니다.

타입 매핑: JPA 기본 타입으로 PostGIS의 Geometry 타입 처리가 불가하여, 엔티티 내 @Column(columnDefinition = "geometry(Point, 4326)") 설정과 Hibernate Spatial 라이브러리를 활용해 매핑 문제를 해결했습니다.
Native Query: ST_SnapToGrid, ST_MakeEnvelope 등 PostGIS 전용 함수는 JPQL에서 지원되지 않아 Native Query로 구현했습니다.
테스트 환경: H2가 Geometry 타입을 지원하지 않아 테스트 실행이 불가한 문제를 H2GIS 라이브러리 도입으로 해결, H2 환경에서도 PostGIS 공간 함수가 정상 동작하는 테스트 환경을 구축했습니다.
